# 학습 여정 — CPU 복습

## 막혔던 지점과 돌파 흐름

### 1. "왜 Central인가"에 효율로 답함
- **내 답** — "여러 부품을 왔다갔다하면 효율이 안 나오니까 중앙 집중"
- **실제 이유** — **역사적 맥락**: 초기 컴퓨터는 연산·제어·저장이 분리된 별도 회로. "Central"은 효율이 아니라 **범용성 확보**를 위한 통합
- **깨달음** — "왜 이 이름인가"는 기능 효율이 아니라 **이전 구조와의 대비**에서 온다

### 2. 기존 문제를 "불편함"으로만 이해
- **내 답** — "배선 바꿔야 하는 게 매번 불편"
- **실제 핵심** — 개별 사례의 불편함이 아니라 **"하나의 기계가 하나의 일만 가능" = 범용성 부재**
- **깨달음** — 문제를 말할 땐 "불편하다"가 아니라 **"무엇이 불가능했는지"** 로 추상화해야 함

### 3. 임시 저장 공간 = 캐시라고 답함
- **내 답** — "CPU 안, 빠른 속도, 작은 용량 → 캐시"
- **놓친 것** — **레지스터** (속성은 맞았는데 이름이 떠오르지 않음)
- **깨달음** — **캐시 vs 레지스터 구분이 본인의 빈틈**
  - **레지스터** — 명령어가 직접 지정해서 읽고 쓰는 공간 (`ADD R1, R2, R3`). ALU가 즉시 꺼내 씀
  - **캐시** — RAM에서 자주 쓰는 데이터를 하드웨어가 **자동으로** 당겨둔 버퍼
- 이 둘을 혼동했던 이유: 속도·위치·용량 속성이 비슷해서

### 4. F-D-E 중 F를 "find"로 추측
- 메모리에서 "가져오다"의 정확한 용어는 **fetch** (find 아님)
- 사소하지만 용어는 정확히 굳혀두는 게 나중에 혼동 줄임

### 5. 코어 개념을 처음 직관적으로 이해
- **내 추론** — "코어 하나 = F-D-E를 굴릴 수 있는 제어장치·ALU·레지스터 세트 하나"
- 이 연결이 CPU 3요소와 멀티코어를 하나로 묶어줌
- **깨달음** — 8코어 = 같은 순간에 명령어 **8개를 진짜 병렬**로 실행. "동시 실행"이 비유가 아니라 물리적 사실

### 6. 스레드 = "명령어 저장공간"으로 착각
- **내 해석** — 스레드는 명령어를 담는 공간
- **실제** — 스레드는 **"실행 흐름의 단위"**. 각 스레드는 "지금 어느 명령어 실행 중 + 레지스터 값들"이라는 **컨텍스트**를 가짐
- **깨달음** — 컨텍스트 스위칭의 "컨텍스트"가 바로 이 **레지스터 값들의 스냅샷**

### 7. 컨텍스트 저장 위치를 캐시로 답함
- **내 답** — "A 상태를 캐시에 저장, B 상태를 레지스터에 올림"
- **정정** — A의 레지스터 값들을 **RAM**에 저장 → RAM에서 B의 값들을 **레지스터**로 복원
- 캐시는 "의도적 저장 공간"이 아니라 하드웨어 자동 최적화 버퍼

### 8. 스레드 풀 사이징은 정확히 맞춤
- CPU 바운드 → 코어 수 정도
- I/O 바운드 → 코어 수보다 많이
- 이건 원래 CPU.md에 있던 내용이라 이미 내재화됨

## 학습 방식 회고

- **소크라테스식 문답이 기대 이상으로 효과적**
- "정리본 읽기"였으면 레지스터 vs 캐시 혼동을 **발견하지 못했을 것**. 질문 덕분에 내 빈틈이 드러남
- 3단계 스캐폴딩 중 **힌트 단계**가 가장 유용 — 답을 주지 않으면서도 방향을 좁혀줌
- 탈출구는 쓰지 않음. 힌트로 충분히 도달 가능했음
