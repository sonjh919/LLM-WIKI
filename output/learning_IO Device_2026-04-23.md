# IO Device 복습 — 막혔던 지점과 깨달음

2026-04-23. Storage에 이어 같은 날 두 번째 세션이자 **컴퓨터부품 5종 마무리**. N=4 학습 여정 노트.

## 매끄럽게 흘러간 부분

- **I/O Device 3분류** (입력·출력·입출력) — 예시(키보드·모니터·NIC) 한 번에 답
- **서버에서 NIC 중요성** — "사용자 외에 서버는 누구와 소통?" 한 줄 힌트에 즉시 도달
- **Thread-per-Connection 한계 — 최강 연결 신호** — "10만 명 동시 접속" 질문에 본인이 스스로 세 가지를 엮어 답:
  - 스레드 × Stack → OOM ([[JVM 메모리 구조]])
  - 스레드 풀로 제한
  - [[컨텍스트 스위칭]] 비용 폭증

  → **이전 4개 세션(CPU·RAM·System Bus·Storage)의 개념이 한 질문에 전부 소환됨**. 세션 간 연결이 정착한 신호.

## 막혔던 지점 ①: I/O 멀티플렉싱 명칭 불명

**내 답**: "이벤트 루프?"

**실제**: I/O 멀티플렉싱(epoll·Selector). 이벤트 루프는 그 위의 상위 패턴.

**깨달음**: **방향은 맞았는데 이름 레이어만 달랐음**. "하나의 스레드가 여러 연결 감시"라는 핵심 구조는 잡았음. 이름은 학습 후 정착.

## 막혔던 지점 ②: 소켓 개념 미정착

**내 답**: "사실 소켓의 정확한 개념을 이해하진 못한 것 같아. 이건 후보로 넣어두고..."

**깨달음**: **세션 중 자기 인식으로 "후보로 빼두자"라고 스스로 분리함**. 이게 곧이어 **학습 후보 백로그 제안으로 연결**. 파생 개념을 세션에서 소화하지 않고 별도로 쌓는 패턴의 첫 사례.

## 막혔던 지점 ③: 5대 부품 조감도의 시작점

**내 반응**: "어디서부터 시작하면 되는거지? 첫 시작을 알아야 할 것 같은데.. 이것도 문제인가?"

**깨달음**: 각 부품은 설명 가능하지만 **한 요청 처리의 흐름으로 엮는 상상**은 아직 따로 연습이 필요. 부품들을 개별로 익혔어도 **조감도는 별도 스킬**. 시각화(canvas)의 필요성이 여기서 드러남.

## 돌파: "학습 후보 백로그" 메타 제안 (본인 발의)

**내가 먼저 제안**: "지금 db처럼 학습하다가 다른 쪽으로 연결지어 새로운 분야로 학습할 수 있는 후보들을 모아두는건 어때?"

**도달한 결정**:
- `output/학습_후보.md` 파일로 관리 (시각화에서 파생 맥락 표현 가능)
- 세션 종료 시마다 업데이트
- CLAUDE.md에 **Branch 프로세스** 추가 (Ingest/Query/Lint/Promote/Reflect와 동급)

**이 제안이 중요했던 이유**: 개별 개념 학습을 넘어 **학습 메타 인프라를 스스로 제안**한 단계. 복습 세션이 개념 외에 학습 시스템 자체를 진화시키는 엔진이 됨.

## 학습 방식 회고 (N=4)

**N=1(CPU)·N=2(RAM)·N=3(System Bus) 대비 달라진 점**:

1. **이전 세션 개념 자동 소환의 성숙** — 이번엔 단일 질문 하나에 JVM 메모리 구조·컨텍스트 스위칭·DMA·버퍼 풀이 동시에 엮여 나옴. 세션 간 연결이 "단발성"에서 "동시 다발성"으로 확장
2. **본인 메타 질문이 학습 인프라로 확장** — N=2·N=3은 개념 확장 질문("JVM이 RAM에 올라가?", "다른 장치 간 버스 있냐?"). N=4는 **학습 시스템 자체에 대한 질문**("후보를 모아두면?") → 학습자 메타 레이어 진입
3. **자기 인식으로 후보 분리** — "소켓 개념 미정착"을 세션 중 스스로 인식하고 후보로 빼둠. 모든 파생 개념을 세션에서 소화하려 하지 않는 절제
4. **포맷 4단 구성 N=4 연속 유효** — 정식화 충분 신호. **스킬화 판단 가능 시점**
