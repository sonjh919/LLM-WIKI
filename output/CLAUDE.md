# output/ — 결과물

위키 지식을 기반으로 생성한 최종 결과물을 저장한다.

## 결과물 유형
- 시각화: canvas, 흐름도, 구조도, 마인드맵
- 학습 계획: 맞춤형 학습 로드맵
- 리마인드 노트: 빠른 복습용 요약

## 규칙
- 결과물은 wiki/ 페이지를 기반으로 생성한다
- 파일명에 유형을 명시한다 (예: `canvas_Spring_전체구조.canvas`, `plan_DB학습계획.md`)
- 특정 형태의 결과물이 반복 생산될 경우 **하위 폴더에 별도 CLAUDE.md**로 세부 가이드 분리 (예: `output/컴퓨터구조 빌드업/CLAUDE.md`)

## 시각화의 목적
- 시각화(canvas, 인포그래픽, 흐름도 등)는 **학습 내용·생각 과정을 효율적으로 복습하여 장기기억으로 전이**시키기 위한 도구
- "예쁜 결과물"이나 "조감 장식"이 아님 — 복습 효율(회상 속도·정착도)이 평가 기준
- 따라서 시각화는 만든 후 끝이 아니라 **반복적으로 다시 보게 되는 형태**여야 함

## 시각화 적용 기준
- 시각화는 **공간적·물리적 관계**가 본질인 주제에만 생성한다 (컴퓨터 부품, 네트워크 토폴로지, 아키텍처 다이어그램 등)
- **개념적 연관**(코딩 원칙, 설계 패턴, 학습 분류 등)은 시각화하지 않는다 — Obsidian graph view가 wikilink로 자동 그래프를 제공하므로 수동 canvas는 중복
- 개념 원칙에 실전 가치를 주려면 조감도가 아닌 **체크리스트·결정 트리·표**를 검토한다
- 페이지가 쌓일 때 "조감이 필요한가?"를 먼저 묻기보다, graph view로 충분한지부터 확인한다

## 시각화 형태 (능동 회상 강제)

목적(복습 → 장기기억)을 달성하려면 "그냥 보는 것"이 불가능한 형태여야 한다. **답 즉시 노출은 이 목적을 무력화**한다.

### 두 종류

**1. 개념 진화형** — 위키 개념 페이지 1개와 1:1 매핑
- 학습 프레임워크 3질문(왜 이 이름 → 기존 문제 → 해결 방법)에 답할 때마다 시각이 한 단계씩 진화
- 답 못 떠올리면 시각 그대로 → 회상 강제
- 적합 개념: "기존 문제 → 해결" 서사가 있는 모든 위키 페이지 (사실상 모든 개념 페이지)

**2. 빌드업형** — 여러 개념이 학습 순서대로 누적되어 조감도가 출현
- 빈 캔버스 → 개념 1 등장·진화 → **페이즈 전환 질문** → 개념 2 등장 → ... → 마지막에 조감도 완성
- 페이즈 전환 질문("지금 상태의 한계는?")이 인과 사슬을 형성 → 조감도가 단순 누적이 아니라 빌드된 결과
- 사용자 학습 궤적 순서를 따라가는 것이 자연스러움

### 학습 프레임워크 ↔ 시각 변화 매핑
- "왜 이 이름?" → 작음 (키워드 강조)
- "기존 문제?" → 큼 (병목·문제 빨갛게, 통증 시각화)
- "해결 방법?" → 가장 큼 (새 요소 등장, 흐름 재배선)

### 안티 패턴
- **클릭 즉시 답 노출** — 능동 회상 무력화. 학습 도구가 아닌 장식이 됨
  (현재 `컴퓨터구조 학습 여정.html`이 해당. 추후 모드 토글 또는 빌드업형으로 재설계 후보)
- **정적 조감만** — graph view나 canvas로 충분, 인포그래픽으로 만들 가치 없음
- **장식적 애니메이션** — 회상과 무관한 시각 효과는 인지 자원 분산만 일으킴
