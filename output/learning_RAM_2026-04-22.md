# RAM 소크라테스식 복습 — 막혔던 지점과 깨달음

2026-04-22. CPU 복습(같은 날)에 이어 두 번째 세션. N=2 시점의 학습 여정 노트.

## 매끄럽게 흘러간 부분

- **RAM 이름의 의미** — Random Access, 임의 위치 동일 속도. 이름 자체에 핵심이 담겨있음을 기억하고 있었음
- **반대 개념** — Sequential Access 바로 떠올림 (카세트테이프까지)
- **RAM 필요성** — "Storage가 느려서" 직관은 바로 나옴
- **휘발성** — 전원 꺼지면 날아감 → Storage 필요성으로 연결

## 막혔던 지점 ①: Storage와 RAM 속도 차이 감

**내가 말한 것**: "몇 배일까? 전혀 감이 안 오는데"

**실제 숫자의 무게**:
- 레지스터 1초 기준 → RAM 100초 → SSD 1일 → HDD 4개월
- HDD와 CPU는 **1천만 배** 차이

**깨달음**: "RAM 왜 필요해?"에 대한 답이 "빠르니까"에서 **"빠르니까 + 그 차이가 극단적이니까"** 로 바뀜. 숫자 없이는 메모리 계층 구조의 필연성이 안 와닿음.

## 막혔던 지점 ②: System Bus 3종 — 이미 복습한 개념인데 흐릿

**내 답**: "주소 버스, 데이터 버스, 플래그 버스?"

**정답**: 주소 버스, 데이터 버스, **제어 버스(Control Bus)**

**배운 것**:
- "플래그"라는 단어 선택은 방향 자체는 맞았음 (read/write 같은 신호를 전달하니까)
- 하지만 **정식 명칭이 제어 버스**임을 까먹고 있었음 → System Bus 페이지 복습이 따로 필요한 신호
- **복습한 개념이 N주 지나면 흐릿해진다** — 소크라테스식 문답이 이걸 드러내는 데 효과적

## 막혔던 지점 ③: JVM 메모리 구조

**첫 반응**: ".class 파일이 올라가지 않을까?" (방향은 맞음 — 메서드 영역)

**힌트 받고 도달**: Stack, Heap

**처음엔 몰랐던 것**:
- **메서드 영역**의 존재와 역할 (클래스 메타데이터·static·상수 풀)
- **공유/비공유 구분** — Heap·메서드 영역은 공유, Stack은 스레드별 별도
- **"JVM도 하나의 프로그램이라 OS에게 RAM을 받는다"** 는 한 단계 위의 관점 (내가 직접 질문해서 명확해짐)

**깨달음**: "JVM 메모리 영역"이라는 말 자체가 **"JVM이라는 프로세스가 OS에서 할당받은 RAM 공간을 내부적으로 어떻게 쪼개 쓰는가"** 의 이야기임. 이 관점이 잡히니 Heap/Stack이 "허공의 개념"이 아니라 "실제 RAM의 일부"로 보임.

## 막혔던 지점 ④: OOM 시나리오 3종

**내 답**: "잘 모르겠네"

**배운 3가지 + OOM 에러 메시지로 영역 구분**:
1. 메모리 누수 — `static List` 계속 add (GC 대상 X)
2. 대용량 쿼리 — `findAll()` 후 페이징 없이 통째로
3. Eviction 없는 캐시 — `HashMap` 무한 put

실무 빈틈. 원리는 알아도 **"어떤 코드가 OOM을 유발하는가"** 의 구체 패턴은 따로 학습이 필요함.

## 막혔던 지점 ⑤: 스레드-Stack 관계

**내 답**: "컨텍스트 스위칭 때 필요한 정보가 스레드에 저장되어 있고, 스레드가 스택에 있어서?" (본인이 "너무 약한 부분" 인정)

**정확한 그림**:
- **스레드마다 자기 Stack이 하나씩** 할당됨 (스레드가 Stack에 "들어있는" 게 아님)
- 이유: 각 스레드가 독립적으로 메서드를 호출하니까 **호출 순서 메모**를 공유할 수 없어서
- 스레드 1000개 × 1MB Stack = 그냥 Stack만 1GB

**깨달음**: "스레드가 Stack에 있다"는 방향이 거꾸로. **"Stack이 스레드에 속한다"** 가 맞는 그림.

## 돌파 ⑥: Java Thread = OS Thread (본인 메타 질문으로 연결)

내가 먼저 질문: "여기서 말하는 스레드는 CPU 복습 때 이야기한 스레드와 같은 의미야?"

**정리**:
- HotSpot JVM에선 **Java Thread 1개 ↔ OS 스레드 1개 (1:1 매핑)**
- CPU 복습의 "컨텍스트 스위칭 대상 스레드" = RAM 복습의 "Stack 먹는 스레드" = **같은 스레드**
- 그래서 **"스레드 많이 만들면 안 되는 이유"가 두 축**으로 설명됨:
  - CPU 측: 컨텍스트 스위칭 오버헤드
  - RAM 측: Stack 누적 소비

**이 질문이 중요했던 이유**: 두 개념을 따로 배웠을 때는 "다른 스레드 아닌가?" 싶은 의심이 생기는데, **같다는 사실이 확인돼야** CPU↔RAM 복습이 하나의 그림으로 묶임.

## 학습 방식 회고 (N=2)

**CPU 복습(N=1) 대비 달라진 점**:
- 탈출구("답 줘") 한 번 사용 (Storage 속도 차이 숫자) → 팩트성 숫자는 바로 주는 게 효율적이라는 신호
- **내가 먼저 메타 질문을 던짐** ("JVM이 RAM에 올라가?", "Java Thread = OS Thread?") → 복습이 깊어지는 지점이 이 질문들이었음
- **이전 복습의 개념(System Bus)이 흐릿**한 것을 확인 → 복습도 감가상각된다는 신호

**learning_*.md 포맷 판단 (N=2)**:
- "막혔던 지점 → 돌파 → 깨달음" 3단 구성이 다시 작동함. 포맷 유지
- **"본인이 먼저 질문한 메타 순간"을 별도 섹션으로 기록**하는 게 도움됨. 이건 CPU 노트엔 없던 축
- N=3까지 가서 포맷 정식화 여부 최종 판단
