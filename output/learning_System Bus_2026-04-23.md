# System Bus 소크라테스식 복습 — 막혔던 지점과 깨달음

2026-04-23. CPU(N=1)·RAM(N=2)에 이어 세 번째 세션. N=3 시점의 학습 여정 노트.

## 매끄럽게 흘러간 부분

- **"Bus" 어원** — "실어나른다"로 비유를 잡음. 방향 맞음(보정 한 줄로 충분)
- **제어 버스 이름 복구** — **지난 세션(2026-04-22)에 "플래그 버스"라고 틀렸던 자리를 이번엔 "제어 버스"로 기억해냄**. 같은 날 쓴 복습 노트·힌트 주기가 재각인한 것으로 추정
- **I/O 바운드 자동 소환** — "CPU 사용률은 낮은데 처리량이 안 나옴" 힌트에 바로 [[CPU]] 복습 때 개념을 꺼냄 → 개념 간 연결 강화 확인

## 막혔던 지점 ①: "기존 문제" 스케일 잘못 잡음

**내 답**: "ALU, 레지스터, 제어장치 등 서로 간에 데이터를 주고받을 수 없어서"

**실제**: System Bus는 CPU **내부** 구성 연결이 아닌 **부품 간**(CPU ↔ RAM ↔ IO ↔ Storage) 연결

**깨달음**: 비슷한 단어("서로 연결")를 써도 **층위**가 다르면 개념이 꼬임. "어느 스케일의 이야기인가"를 먼저 특정하는 습관이 필요.

## 돌파 ①: "왜 3개 버스로 분리했는가" (본인 메타 질문)

**내가 먼저 질문**: "주소값이랑 제어 신호를 거의 같이 주는데, 그럼 제어 버스가 나뉜 이유가 뭐지?"

**도달한 답**:
- 정보의 **성격이 다름** — 주소(위치), 데이터(내용), 제어(행위 지시)
- 물리적으로 분리하면 **해석 오버헤드 제거** ("지금 이 비트는 뭐다"를 매번 파싱할 필요 없음)
- 용도별 **폭 최적화** — 주소/데이터는 32·64비트, 제어는 1비트 신호 묶음

**이 질문이 중요했던 이유**: 그냥 "3개로 나뉘어 있다"를 외우는 것과, **왜 나눴는지의 설계 철학**을 이해하는 건 질적으로 다른 학습. 메타 질문 없이는 이 깊이에 도달 못 함.

## 막혔던 지점 ②: 제어 버스가 언제 필요한지

**내 답**: "CPU가 RAM에 주소를 주고 데이터를 받아. 제어 신호는 언제 필요한거지?"

**힌트**: "RAM 입장에서 주소 100번지가 읽기인지 쓰기인지 주소만 봐선 모름"

**깨달음**: 제어 버스는 "부가적 조율" 같은 막연한 역할이 아니라, **주소만으로는 구별 불가능한 정보를 명시**하는 역할. 구체 장면(읽기 vs 쓰기)이 있어야 필요성이 와닿음.

## 돌파 ②: "다른 장치끼리 이어지는 버스 있냐" (본인 메타 질문 → DMA)

**내가 먼저 질문**: "버스는 cpu랑 다른 장치 사이만 있는거야? 다른 장치끼리 이어지는 버스는 없어?"

**도달한 답**: **DMA (Direct Memory Access)**
- 옛날엔 CPU가 바이트 단위로 NIC ↔ RAM 복사를 직접 해야 했음
- DMA 컨트롤러가 버스를 빌려 IO ↔ RAM을 직접 연결
- CPU는 "시작해"/"끝났어" 알림만 처리, 그 사이 다른 연산 수행

**이 질문이 중요했던 이유**: "버스는 CPU 중심"이라는 암묵적 전제를 **스스로 의심**한 게 DMA 개념의 자연스러운 진입점이 됨. 답을 듣고 끝이 아니라 **다음 질문을 낳는 복습 모드**.

## 막혔던 지점 ③: I/O 바운드 원인 — 버스 폭이 좁아서?

**내 답**: "cpu와 네트워크 장치 간 데이터 버스의 폭이 좁아서?"

**보정**: 버스는 이미 충분히 빠름. 진짜 원인은 **IO 장치 자체의 속도** (SSD 100μs·HDD 10ms·네트워크 ms).

**깨달음**: "병목"이라는 같은 단어 안에도 **층위 구분**이 필요함:
- 버스 자체 병목 — 특수 영역(GPU·NVMe 극한)에서만
- IO 장치 병목 — 일반 서버의 실질 병목

이 구분이 없으면 "버스 = 병목"으로 잘못 연결됨. 뒤이어 **비동기/논블로킹**이 "IO 기다림 처리 모델"로서 왜 답인지 와닿음.

## 학습 방식 회고 (N=3)

**N=1(CPU)·N=2(RAM) 대비 달라진 점**:

1. **복습 감가상각의 역방향 신호** — 지난 세션에 흐릿했던 System Bus 3종이 이번엔 **제어 버스까지 복구됨**. 하루 단위 복습 주기가 짧아서일 수도, learning 노트가 재각인했을 수도 있음. **N주 후 다시 확인 필요**
2. **본인 메타 질문이 2개** (세션당 2개, RAM 세션과 같은 빈도) — 그리고 모두 핵심 개념(분리 이유, DMA)으로 자연 유도. **메타 질문이 이 학습 방식의 핵심 엔진**이라는 신호
3. **이전 세션 개념 자동 소환** — I/O 바운드(CPU 복습)가 힌트 한 줄로 즉시 나옴. 세션 간 연결이 머릿속에서 일어나고 있음

**learning_*.md 포맷 판단 (N=3)**:
- "매끄럽게 흘러간 부분 → 막혔던 지점 → 돌파(메타 질문) → 회고" 4단 구성 **3회 연속 유효**
- **"본인 메타 질문" 섹션**은 N=2에서 추가한 이후 N=3에서도 핵심 가치 입증
- **포맷 정식화 가능 신호** — N=3 도달. 다음 복습 세션에서 스킬/템플릿화 검토

**다음 신호**:
- 복습도 감가상각된다는 관찰(RAM 세션)과, 복구도 가능하다는 관찰(System Bus 세션)이 충돌. **복습 간격·재각인 조건**에 대한 실험이 필요해질 수 있음
- 남은 부품 2개(IO Device, Storage)도 같은 방식으로 복습 후 시각화·연결 종합 예정
