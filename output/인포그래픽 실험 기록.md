# 인포그래픽 output 실험

output/ 결과물로 HTML 인포그래픽(인터랙티브, 애니메이션) 시도 기록.
형태가 확정되기 전 실험 과정과 학습을 남기는 문서.

## 실험 이력

### N=1: `컴퓨터부품 인포그래픽.html`
- 형태: SVG 메인보드 위에 5대 부품 배치, 부품 클릭 시 사이드 패널로 학습 정보 표시
- 평가: 종합 조감형 / 클릭 즉시 답 노출

### N=2: `컴퓨터구조 학습 여정.html`
- 형태: N=1 위에 학습 세션 뱃지(메타 질문 호버) + 9단계 요청 흐름 재생 + 학습 후보 백로그 드로어 추가 (3층)
- 평가: 종합 조감형 / 클릭 즉시 답 노출 (메타 질문 호버만 부분적으로 질문 형태)

## 발견한 미스매치 (2026-04-27)

**문제**: 두 인포그래픽 모두 "클릭 즉시 답 노출" 구조라 **능동 회상이 강제되지 않음**. 학습 모드는 소크라테스식 문답인데 인포그래픽은 그냥 읽게 됨 → 장기기억 전이 효율 ↓

**원인 분석**:
- 두 개 모두 **종합 조감형**(공간 배치 본질). 종합 조감형은 "초기 상태 → 진화"의 서사가 어색해서 능동 회상 패턴을 자연스럽게 못 입힘
- 단일 개념(CPU·RAM·DMA 등)은 위키 페이지가 이미 "왜 이 이름 → 기존 문제 → 해결" 프레임워크로 정리돼 있어서 진화형 시각화에 자연스럽게 맞음

## 결정한 새 방향: 빌드업형 (N=3)

**형태**: 빈 캔버스에서 시작 → 학습 궤적 순서대로 개념이 등장·진화 → 마지막에 조감도가 출현

```
P1: CPU 등장 → 3단계 진화
P2: 페이즈 전환 질문 ("지금 상태의 한계?") → RAM 등장 → 진화
P3: System Bus → 진화
P4: Storage → 진화
P5: IO Device → 진화
P6: 9단계 요청 흐름 활성화 (조감도 완성)
P7: (보너스) 승격 개념 오버레이 — DMA·컨텍스트 스위칭·JVM 메모리
```

**왜 강력한가**:
1. 매 학습이 "내가 빌드한 한 조각"으로 누적 → 소유감
2. 페이즈 전환 질문이 부품 간 인과 사슬을 형성 (왜 다음이 필요한가)
3. 복습 시 매번 처음부터 빌드하면서 회상 강제
4. 진화형 + 조감형이 한 형태로 통합

**프로토타입 범위**: P1+P2(CPU + RAM 등장)까지만 먼저. 형태 검증 후 P3~ 확장.

## 기존 두 인포그래픽 처분

폐기 X. 추후 둘 중 하나로 재설계 후보:
- 모드 토글 (질문 모드 ↔ 참조 모드)
- 빌드업형으로 통합 흡수

빌드업형 N=1 검증 후 결정.
