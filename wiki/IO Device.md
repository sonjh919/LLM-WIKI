## 왜 이 이름인가

I/O Device = Input/Output Device (입출력 장치). **컴퓨터 내부(CPU·RAM·Storage)와 외부 세계의 경계**에 서서 데이터를 "들여오고(Input) 내보내는(Output)" 장치군.

경계의 상대는 두 종류:
- **사람 ↔ 컴퓨터** — 키보드·마우스·모니터
- **컴퓨터 ↔ 컴퓨터** — NIC (네트워크 카드). 서버 입장에선 가장 중요한 I/O

[[Storage]]도 엄밀히는 I/O의 일종이지만 "보조기억장치"로 따로 분류. 나머지를 I/O Device로 묶음.

## 기존 문제

[[CPU]]·[[RAM]]·[[Storage]]만으로는 **외부와 소통 불가**. 사용자 입력·화면 출력·네트워크 요청·응답 — 전부 불가능. "컴퓨터 내부와 외부를 연결하는 통로가 필요하다."

## 어떻게 해결하는가

I/O Device가 [[System Bus]]를 통해 CPU·RAM과 데이터를 주고받음.

- **입력** — 키보드, 마우스, 마이크
- **출력** — 모니터, 프린터, 스피커
- **입출력** — 네트워크 카드(NIC), USB, 터치스크린

### I/O가 느린 이유와 대응

I/O Device는 CPU 대비 극도로 느림 (네트워크 ms, 디스크 μs~ms vs CPU ns). CPU가 I/O 완료를 매번 동기 대기하면 대부분의 시간을 낭비.

대응 메커니즘 두 축:
- **인터럽트** — 완료 시점에 CPU에 알림. 폴링 제거
- **[[DMA]]** — CPU를 거치지 않고 I/O ↔ RAM 직접 전송. CPU는 "시작·끝" 명령만

## 백엔드 개발에서의 활용

### Thread-per-Connection vs I/O 멀티플렉싱

I/O 대기 문제의 서버측 대응 두 패러다임.

**Thread-per-Connection (전통 방식)**:
```
소켓 1 → 스레드 1 (blocking)
소켓 2 → 스레드 2 (blocking)
소켓 N → 스레드 N
```
- OS가 [[컨텍스트 스위칭]]으로 블로킹된 스레드를 교체 → CPU는 놀지 않음
- **한계**:
  1. 스레드 × Stack → [[JVM 메모리 구조]] 기준 스레드당 ~1MB. 10만 연결이면 OOM
  2. 스레드 풀로 제한하면 대기 연결이 쌓임
  3. 컨텍스트 스위칭 자체의 오버헤드

**I/O 멀티플렉싱**:
```
소켓 1 ─┐
소켓 2 ─┼─→ 스레드 1개가 "준비된 것"만 감시·처리
소켓 N ─┘
```
- OS가 "어느 소켓에 데이터 도착" 알림 → 스레드는 준비된 것만 처리. I/O 대기로 블로킹되지 않음 (논블로킹)
- **구현체**: Linux `epoll`, BSD `kqueue`, Java NIO `Selector`
- **위에 얹은 프레임워크**: **Netty** → **Spring WebFlux**, Node.js 이벤트 루프도 같은 원리
- 수백~수만 연결을 소수 스레드로 처리

### 네트워크 I/O의 중심성

서버의 주된 I/O는 NIC. HTTP 요청·응답, DB 커넥션, 외부 API, 서버 간 통신 모두 NIC 경유. 백엔드 서버가 대부분 I/O 바운드인 이유.

### 디스크 I/O와의 결합

파일 업로드·다운로드, 로그 쓰기는 NIC ↔ [[Storage]]의 결합 경로. `java.nio`의 Non-blocking I/O, [[DMA]] 기반 `sendfile()`/zero-copy가 이 맥락에서 성능 핵심.
