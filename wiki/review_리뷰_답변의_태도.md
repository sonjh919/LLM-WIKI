# review: 리뷰 답변의 태도 — 수긍에서 공동 고민으로

> 이 페이지는 **코드가 아닌 답변**에 대한 원칙이다. 기술 원칙이 "무엇을 코딩하는가"라면, 여기는 "리뷰어와 어떻게 대화하는가".

## 이 답변이 왜 문제인가

### 패턴 1: 단순 수긍 — "네 맞습니다, 수정하겠습니다"

```
리뷰어: final 키워드를 붙이는 게 좋겠어요.
저자: 네 맞습니다 수정하겠습니다.
```

- 리뷰어의 **의도·근거**가 내 것이 되지 못함 → 다음 PR에서 다시 지적받음
- 수정은 했지만 **"왜 그렇게 하는가"** 를 언어화하지 못함 → 유사 상황에서 재적용 실패

### 패턴 2: 방어적 변명 — "그럴 이유가 있었습니다" (관성을 감춤)

```
리뷰어: GetLottoDto라는 이름은 어때요?
저자: Spring에서 관례상 get prefix를 쓰기에 익숙해서요...
     (사실은 IDE 자동 완성을 검토 없이 둔 관성)
```

- 변명은 **학습의 벽을 세움** — 리뷰어도 더 깊이 묻기 어려워짐
- 관성(자동 생성·과거 습관)을 감추면 같은 관성이 다음에 또 나옴

### 패턴 3: 근거 없이 반론만 — "저는 이게 맞다고 생각합니다"

```
리뷰어: 컨트롤러에 도메인 로직이 들어간 것 같네요.
저자: 이건 도메인 로직이 아니라고 생각합니다.
     (왜 그렇게 판단했는지 설명 없음)
```

- 근거가 빠진 반론은 **대화를 단절**시킴
- 판단의 이유가 말로 나오지 않으면 **틀렸을 때도 수정 지점이 안 보임**

### 패턴 4: 미해결 트레이드오프를 숨김

```
저자: protected를 제거하고 private으로 바꿨습니다. 해결!
     (실제로는 getter 재정의 중복이라는 새 불편이 생겼지만 언급 X)
```

- 표면만 해결한 척 → 리뷰어는 **추가 피드백 지점을 놓침**
- 학습자로서 **"여전히 고민되는 부분"** 을 숨기면 그 부분은 영영 해결 안 됨

## 대안 — 답변의 성장 3단계 (실제 관찰됨)

### 1단계: 수긍 + 역질문 (로또 PR, 2025-02)

```
리뷰어 (Rok93): 파라미터에도 final을 붙이는 게 좋아요.
저자: 아 그렇군요. 혹시 final은 왜 붙이는 건가요?
     (수정은 하되, 이유를 물어 내 것으로 만듦)
```

```
저자: 클래스가 2개이니 당연히 팩토리도 2개를 만들어야 한다는
     생각이 무의식에 있었던 것 같아요!
     (관성·무의식을 솔직하게 드러냄)
```

- **수긍만 하지 않고 역질문** — 원칙의 근거를 받아냄
- **관성을 변명 없이 인정** — "프리코스 습관", "자동 완성 흔적"

### 2단계: 근거 있는 반론 (블랙잭 PR, 2025-03 초)

```
리뷰어 (ddaaac): 왜 도메인이 뷰를 몰라야 하는가? 그 이유가
                가독성을 해치면서까지 지킬 만큼 중요한가?
저자: 이 로직은 도메인 관심사라 생각했습니다. 이유는...
     (판단의 근거를 1~2단락으로 전개, 여전히 역질문도 병행)
```

- **의식적 선택의 근거를 언어화** — 왜 이 길을 골랐는지 말로 풀어냄
- 반론이 있되 **열려 있음** — "혹시 설계 꿀팁이 있나요?"로 대화 유지

### 3단계: 공동 고민 — 미해결을 드러내는 탐구 (장기 PR, 2025-03 말)

```
저자: protected를 제거하고 private + getter 재정의로 바꾸었습니다.
     각 객체의 책임이 명확히 분리된 점은 좋지만,
     getter를 추상 메서드로 정의하고 하위마다 재정의해야 하는
     불편함이 생겼어요.
     이 설계에 대해서는 어떻게 생각하시나요?
     이전보다 좋은 설계라고 생각되시나요? 😊
```

- **설계 근거를 단락 단위로 서술**
- **미해결 트레이드오프를 인정** — "새 불편이 생겼다"
- **리뷰어를 답변자 → 공동 탐구자**로 전환

## 적용 기준

### 답변 작성 전 스스로 묻기
1. **"나는 왜 그렇게 코딩했는가?"** — 무의식적 선택이면 그 관성을 인정
2. **"리뷰어가 왜 이 질문을 했는가?"** — 피드백의 의도를 먼저 짐작
3. **"수정 후에도 남는 불편/고민이 있는가?"** — 있으면 드러냄
4. **"내가 다른 선택을 한다면 근거는?"** — 반론이라면 근거와 함께

### 질문형 리뷰(ddaaac·seovalue 스타일)에 대응하기
- "왜 그렇게 했나요?"는 **판단 근거를 언어화하라는 요청**이지, 정답 확인이 아님
- 리뷰어가 답을 주지 않는 건 **"Owner는 당신"** — 자기 선택을 방어하라는 신호
- 근거를 쓰면서 **스스로 판단의 약한 지점을 발견**하는 게 목적

### 지시형 리뷰(Rok93 스타일)에 대응하기
- 수정만 하고 끝내지 말고 **"왜 그 방식이 더 나은지"** 물을 것
- 리뷰어가 답을 주면 그 답을 **내 언어로 다시 쓰기** → 기억으로 전환

### 지양할 답변 패턴
- 단순 수긍 (`"네 수정하겠습니다"`만)
- 근거 없는 반론 (`"저는 이게 맞다고 생각합니다"`만)
- 방어적 변명 (관성을 포장)
- 표면 해결 + 새 불편 은폐

## 흔한 반론과 답

| 반론 | 답 |
|---|---|
| "답변이 길어지면 리뷰어도 부담" | 길이가 문제가 아니라 **신호 밀도**. 근거가 없으면 짧아도 부담. 근거가 있으면 길어도 가치 |
| "모르는 걸 드러내는 게 창피" | 리뷰어는 **완성품이 아니라 학습자**를 보는 중. 솔직한 고민이 더 도움 되는 피드백을 끌어냄 |
| "근거를 쓰다 보면 설득하고 싶어진다" | 설득보다 **탐구**. "이 판단이 약한 지점은 어디인가" 질문을 답변에 포함 |
| "질문형 리뷰에 질문으로 답하면 책임 회피 같다" | 질문도 **자기 답변 뒤에** 붙일 때만 유효. "이렇게 판단했다, 여기가 약점 같다, 어떻게 생각하는가" |

## 3개 PR에서 관찰된 성장 궤적

| 시점 | PR | 리뷰어 스타일 | 내 답변 태도 |
|---|---|---|---|
| 2025-02 | 로또 | 지시형 (Rok93) | 수긍 + 역질문 + 관성 인정 |
| 2025-03 초 | 블랙잭 | 질문형·철학적 (ddaaac) | 근거 있는 반론 + 여전히 역질문 |
| 2025-03 말 | 장기 | 질문형·위임 (seovalue) | 단락 근거 + 미해결 인정 + 공동 고민 |

**핵심 변화**: 답변의 무게중심이 **"정답 맞추기" → "같이 탐구하기"** 로 이동

## 학습 노트 (이 변화는 왜 일어났나)

- **1→2단계 전환 지점**: ddaaac의 질문형 리뷰가 기폭제
  - "왜 그래야 하는가"를 끊임없이 되물어 근거 언어화 강제됨
  - "단일 정답 없음, 직접 구현해보고 비교하라" — 방어 대신 탐구로 전환
- **2→3단계 전환 지점**: seovalue의 "Owner는 링크" 발언
  - 리뷰어가 판단 위임 → 내 선택의 근거를 먼저 전개해야 대화가 성립
  - 미해결을 숨기면 다음 대화가 얕아진다는 걸 체감
- **미수확 원칙** (앞으로 의식할 것):
  - **"답변 = 설득"이 아니라 "답변 = 자기 판단의 외부화"**
  - **관성을 빠르게 이름 붙이기** ("프리코스 습관", "IDE 자동 생성")
  - **새로 생긴 불편을 스스로 먼저 말하기**

## 이 원칙이 AI와의 대화에도 적용되는가

적용됨:
- AI에게 질문할 때도 **내 판단 근거를 먼저** → AI가 약한 지점을 찾기 쉬움
- "수정해줘" 대신 **"이렇게 했는데 이 부분이 애매하다, 어떻게 봐?"** 가 더 풍부한 답 끌어냄
- AI가 제안할 때 **수긍만 하지 말고 "왜 이게 더 나은가"** 를 묻기 — 내 것으로 만들기

세컨드 브레인 비전과 직접 연결: **학습 파트너와 상호작용하는 법 자체가 자산**.
