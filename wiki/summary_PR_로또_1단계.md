## 주요 학습 포인트 (카테고리별)

### 1. 기능 명세와 커밋 단위
- 기능 구현 전 **README.md에 체크리스트로 기능 목록 정리** → 이 단위로 커밋
- 목적: 요구사항 정의가 선행되고, 커밋 메시지·단위가 자연스럽게 깔끔해짐

### 2. 네이밍 — 의도가 드러나게
- **메서드**: `getXXX/setXXX` 지양, **행위**가 드러나는 이름 (`buyLottos`, `calculateProfit`)
- **클래스**: `Main` → `Application/LottoApplication`, `Input` → `InputMessage`
- **팩토리 메서드**: `from()` → `generate()/create()` (생성 행위 명시)
- **도메인 이름**: `Amount` → 의미가 드러나는 이름 (너무 범용적)
- **DTO**: Prefix `Get`은 불필요 (HTTP 메서드 prefix 습관이 잘못 이식됨)
- **필드명**: `bonus` → `bonusNumber` (혼동 방지)
- **파라미터명**도 의미 전달 중요

### 3. 타입 선택 — 도메인에 맞게
- 금액처럼 숫자로 다룰 값은 **처음부터 숫자형**으로 입력받기 (String → int 변환 최소화)
- "프리코스 때 문자열 라이브러리를 썼던 관성"이 실수 원인

### 4. `final` 키워드의 목적
- 재할당 없는 필드엔 `final` → **언어 단에서 변경을 막는 안전장치**
- 메서드 파라미터에도 적용
- 핵심 관점: **"붙이라고 했어"가 아니라 실수 방지**
  - 팀 규모가 커지거나 동료가 실수할 때 컴파일 에러로 빠르게 차단
  - 신경 쓰지 않아도 되는 것이 많을수록 인지 부하 감소

### 5. 객체 생성 패턴
- **완전한 상태로 생성**: 생성 후 일일이 값 추가하지 말고 생성자 파라미터로 주입
  ```java
  // bad
  Lottos lottos = new Lottos();
  lottos.add(lotto1); ...
  // good
  Lottos lottos = new Lottos(List.of(...));
  ```
- **생성자는 생성만**: 파싱(`String → List<Integer>`)·검증은 외부 또는 정적 팩토리 메서드에서
- **정적 팩토리 메서드** 선호 (검증 분리, 이름 있는 생성)

### 6. DTO 변환의 의존 방향
- 도메인 객체가 DTO 변환 메서드를 갖는 대신, **DTO가 도메인에 의존해서 생성**
- 구현: DTO 클래스의 정적 팩토리 메서드 (`LottosDto.from(lottos)`)
- 이유: 도메인은 DTO를 몰라야 (의존 역전)

### 7. 단위 테스트
- UI 제외 모든 로직에 단위 테스트
- `@DisplayName`으로 의도 설명
- 테스트 네이밍도 유지보수 대상 — 사소하게 다루지 말기
- **다양한 입력값으로 테스트 → 버그 발견**

### 8. 불필요한 코드 제거
- 중간 변수 선언 최소화 (`return` 한 줄로 충분하면)
- 불필요한 줄바꿈 제거
- **매직 넘버는 상수화**

### 9. 캡슐화 / 접근 제한자
- 클래스 내부에서만 쓰이는 상수는 `private`
- `protected`는 상속/캡슐화 트레이드오프 고려해서 신중히
- 컬렉션 반환 시 **방어적 복사** (`new ArrayList<>(numbers)`)

### 10. static vs instance
- 인스턴스 상태(`this.prize`)를 쓰는데 static이면 불일치
- 인스턴스 상태 활용 시 instance 메서드로

### 11. 패키지 구조
- 도메인 로직(객체 생성 포함)은 **domain 패키지** 내부에 두는 것이 자연스러움
- `global.factory`로 분리한 건 어색 — 로또 생성도 도메인 로직

### 12. 상속 vs 합성
- 상속 시 필드가 `protected`로 열려 캡슐화 깨짐, 결합도 ↑
- → **합성** (`WinningLotto`가 `Lotto`를 필드로 보유)이 더 유연
