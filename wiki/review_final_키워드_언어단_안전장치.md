# review: `final` 키워드 — 언어 단의 실수 방지 안전장치

## 이 코드가 왜 문제인가

```java
public class Amount {
    public static final int LOTTO_PRICE = 1000;

    private int amount;   // final 없음

    public Amount(String inputPrice) { ... }
}
```

- 설계 의도상 `amount`는 **생성자 이후 변경되지 않는 값**
- 하지만 언어는 그 의도를 모름 → 누구든 `this.amount = ...` 로 실수·의도 외 변경 가능
- 본인이 혼자 관리할 땐 문제 없어 보이지만, **팀·시간이 늘어날수록** 의도 외 변경이 새어나옴
  - 컴파일러가 막아주지 않으면 런타임 혹은 장애로 발견됨

## 대안: `final` 로 컴파일 타임에 차단

```java
public class Amount {
    private final int amount;

    public Amount(final String inputPrice) { ... }
}
```

- **재할당 시도 = 컴파일 에러** → 실수가 런타임까지 가지 않음
- 의도("이 값은 한 번 할당되면 변하지 않는다")가 **코드에 명시적으로 드러남**
- 리뷰어·미래의 자신이 "이 필드가 변할 수 있나?"를 고민하지 않아도 됨 → 인지 부하 ↓

## 적용 기준

### 붙인다
- **필드**: 한 번 할당된 뒤 변경되지 않는 필드 (대부분의 도메인 객체 필드)
- **메서드 파라미터**: 메서드 내부에서 재할당하지 않는 한 (거의 항상)
- **지역 변수**: 재할당 없는 값

### 붙이지 않는다
- 의도적으로 **변경이 필요한** 상태 (mutable 컬렉션, 상태 머신 등)
- 팀 컨벤션에서 명시적으로 배제한 경우

### 애매할 때의 판단
- 질문: "이 값이 나중에 바뀌어야 할 이유가 있나?"
- 답이 "없다"면 붙이기. 나중에 바뀌어야 한다면 그때 제거 + 변경 이유를 주석·네이밍으로 드러내기

## 흔한 반론과 답

| 반론 | 답 |
|---|---|
| "지금까지 실수한 적 없다" | 개인·소규모 경험에선 안 보임. 팀이 커지고 코드가 오래될수록 차이가 누적된다 |
| "코드가 길어진다" | 길이보다 **의도 명시**와 **실수 방지** 가치가 크다 |
| "붙이라고 해서 붙인다" | **가장 나쁜 마인드**. 목적(실수 방지)을 이해해야 경계 판단이 가능 |

## 학습 노트 (왜 놓쳤나)

- "지금까지 final 없어도 이슈 없었다"는 **경험에 의한 과소평가**
- 리뷰어 답변의 핵심: **"문제를 겪지 않았다" ≠ "문제가 없다"**
  - 혼자·소규모에서 의식적으로 프로그래밍하면 문제가 표면에 안 드러남
  - 시스템 규모·팀 크기·유지보수 기간이 늘어날수록 안전장치의 가치가 드러남
- 추출 원칙: **신경 쓰지 않아도 되는 것이 많을수록 복잡한 문제에 집중할 수 있다**
  - `final`은 "한 번만 할당됨"을 언어가 보장해 줌 → 나는 다른 걸 생각할 수 있음
  - 같은 원리: immutable 객체, 타입 시스템, 테스트 자동화 등
