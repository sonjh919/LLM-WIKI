## 주요 학습 포인트 (이 PR에서 새롭게 드러난 것)

### 1. 추상화의 목적 — **요구사항을 코드로 녹이기**
- 잘못된 이유: "중복 제거", "변경 가능성 대비" → 흔히 떠올리지만 본질 X
- 본질: **요구사항을 메시지 흐름으로 표현**하는 것
- 예시:
  > "첫 카드를 공개할 때, 딜러는 한 장, 플레이어는 두 장을 공개한다"
  → 외부가 `"첫 공개 카드 알려줘"` 메시지를 던짐 → `Participant` 추상 메서드로 묶음 → 각 구현체가 자율 응답
- `Participant#getFirstCards`만 봐도 "딜러와 플레이어의 첫 공개 카드가 다르다"는 요구사항이 드러남

### 2. 현실 세계 ≠ 객체 세계 (책임 분배)
- "객체지향의 사실과 오해" 인용: 현실에서 컵은 수동이지만 객체 세계에선 컵도 능동일 수 있음
- 예: `BlackJack`이 `Deck`을 가지면 시그니처가 자연스러워짐 → 딜러가 가질 이유가 없음
- 교훈: **현실을 그대로 반영할 필요 없다**. 객체의 책임은 객체 세계 안에서 판단

### 3. 함수형 인터페이스의 트레이드오프
- 내 선택: `drawPlayers(Function<Player, Boolean>, Consumer<Player>)` — 도메인이 뷰를 모르게
- 장점: 도메인 순수성, 뷰 결합도 ↓
- 단점: **시그니처만으로 책임이 드러나지 않음** → 가독성 타격
- ddaaac의 근본 질문: "왜 컨트롤러에 도메인 로직이 들어가면 안 되는가? 그 이유가 가독성을 해치면서까지 지킬 만큼 중요한가?"
- 결론: 단일 정답 없음. **직접 두 방식 다 구현해보고 비교하는 것**이 학습 방법

### 4. 도메인 요구사항의 모호성·해석
- "딜러는 16 이하면 1장 더 받고 17 이상이면 안 받는다" → 한 장만 더? 17 이상이 될 때까지 계속?
- 요구사항만으로 완전히 설명 안 됨
- **명확하지 않은 요구사항을 명확하게 만드는 것도 개발자의 역량**
- 도메인 공부(블랙잭 공식 룰)가 선행되어야 함

### 5. 테스트용 getter의 판단 기준
- 내 고민: 테스트 때문에 getter를 만드는 게 맞나?
- ddaaac 답: **테스트도 클라이언트 코드**. 클라이언트가 필요한 건 객체의 책임일 수 있음
- 핵심 원칙: **"getter로 값을 꺼내서 로직을 구현하지 않는다"** — getter의 존재가 문제가 아님

### 6. 상수 중복의 해결 — 객체로 추출
- 문제: `BLACKJACK_NUMBER = 21`이 Player, MatchResult 양쪽에 중복
- `public`으로 열기 싫다면 → **Score 객체로 추출**해서 책임을 분리
- 교훈: 상수가 중복된다는 건 **책임이 흩어져 있다**는 신호

### 7. 일반 클래스 상속 지양
- 단순 재사용 목적이라면 **조합(composition)이 더 안전**
- 추상 클래스도 "특별한 케이스가 아니라면" 신중히

### 8. Hand vs Deck의 책임 분리
- 나의 오류: Card의 일급 컬렉션으로 `CardDeck` 하나에 몰아넣음
- 리뷰어 지적: **Deck의 책임 ≠ Hand의 책임** (뽑기 vs 들고 있기)
- 설계 관점 배움: 상태부터 설계 X → **메시지부터 설계** → 공통 상태·행동만 클래스로 묶기 (오브젝트 책)

### 9. 테스트 given은 프로덕션과 동일하게
- 내 실수: given에서 진행 순서를 임의로 설정 → 맹점 발생
- 원칙: 테스트의 given은 **프로덕션의 진행 상황 그대로**

### 10. 학습 방법론 — **직접 구현해서 비교하라**
- 상상만으로 "이 방식이 맞다/아니다" 판단하지 말 것
- 리뷰어가 함수형 방식 대신 for/while 방식으로도 구현해보라고 권하는 이유
- "별로라 생각했는데 막상 구현하면 훨씬 잘 읽히는 케이스도 많다"
