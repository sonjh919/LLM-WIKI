## 왜 이 이름인가

RAM = Random Access Memory (임의 접근 메모리). 데이터가 저장된 위치와 상관없이 **어느 주소든 동일한 속도로 접근**할 수 있어서 "Random Access"라는 이름이 붙음.

- **반대 개념: 순차 접근(Sequential Access)** — 카세트테이프·LTO 테이프처럼 앞에서부터 훑어가야 뒤 데이터에 닿는 방식. 100번째 데이터 읽고 1번째 읽으려면 되감기 필요.
- **왜 "임의 접근"이 중요한가** — 프로그램 실행 중 CPU가 다음에 어느 주소를 읽을지 미리 알 수 없음. 조건문·배열 접근으로 메모리 곳곳을 뒤죽박죽 읽음. 어느 주소든 똑같이 빠르니까 CPU가 예측 없이 자유롭게 실행 가능.

## 기존 문제

CPU는 매우 빠르지만, [[Storage]]는 느리다. 속도 차이가 극단적이다:

| 저장소 | 접근 시간 | 레지스터 대비 |
|--------|----------|--------------|
| 레지스터 | ~1 ns | 1배 |
| L1 캐시 | ~1 ns | 1~3배 |
| L2 캐시 | ~4 ns | 4배 |
| L3 캐시 | ~10 ns | 10배 |
| **RAM** | ~100 ns | **100배** |
| SSD | ~100 μs | **100,000배** |
| HDD | ~10 ms | **10,000,000배** |

> 레지스터 접근을 "1초"로 치면, RAM은 100초(1분 40초)·SSD는 1일·HDD는 **4개월**.

CPU가 매번 Storage를 직접 읽으면 대부분을 기다림에 쓴다. "현재 실행 중인 프로그램과 데이터만 빠른 곳에 올려놓으면 되지 않을까?" → RAM.

## 어떻게 해결하는가

RAM은 현재 실행 중인 프로그램과 데이터를 임시로 저장하는 **주기억장치**. CPU와 RAM 사이는 [[System Bus]]로 연결되어 있고, CPU는 RAM에 올라온 데이터만 처리할 수 있다.

**RAM의 3가지 특성**:

| 특성 | 의미 | 대비 |
|------|-----|------|
| **임의 접근** | 어느 주소든 동일 속도 | 테이프: 순차 접근 |
| **휘발성** | 전원 꺼지면 데이터 소실 | Storage: 비휘발성 |
| **고속·저용량** | 빠르지만 비싸고 작음 | Storage: 느리지만 싸고 큼 |

이 세 특성의 조합이 **"메모리 계층 구조"** 를 만든다. 빠른 작업 공간(RAM)에서 처리하되, 꺼져도 남아야 할 것은 Storage에 내려둔다.

## 백엔드 개발에서의 활용

### RAM 위에서 돌아가는 애플리케이션

JVM도 하나의 프로그램이다. OS에게서 RAM을 할당받아 실행되고, 내부적으로 Heap·Stack·메서드 영역으로 쪼개 쓴다. 자세한 내용·OOM·옵션 → [[JVM 메모리 구조]]

### 동시 요청 처리와 RAM 소비

[[JVM 메모리 구조]]에서 각 스레드는 자기만의 Stack(약 1MB)을 갖는다. 그래서 **동시 요청이 늘수록 RAM 소비가 선형 증가**한다.

```
동시 요청 1000개 → 스레드 1000개 → Stack만 1GB
```

여기에 [[컨텍스트 스위칭]] 오버헤드(CPU 낭비)까지 겹쳐 **"스레드를 무한정 늘리면 안 되는 이유"** 의 두 축이 된다 — **RAM 축과 CPU 축**.

### 인메모리 캐시

Redis, Caffeine 같은 캐시가 빠른 이유는 디스크가 아닌 **RAM에 데이터를 저장**하기 때문. 속도 계층 표에서 RAM과 SSD는 **1000배 차이**라 캐싱 효과가 압도적이다.
