# wiki/ — AI가 관리하는 위키

이 폴더는 AI가 작성하고 유지하는 지식 베이스다. 사용자는 읽고, AI가 쓴다.

## 페이지 유형
- **개념 페이지** — 하나의 기술 개념을 설명 (예: `트랜잭션.md`)
- **요약 페이지** — 원본 소스 하나를 요약 (예: `summary_Spring_DI.md`)
- **PR 요약 페이지** — GitHub PR 하나를 요약 (예: `summary_PR_로또_1단계.md`). 프레임워크: "PR 개요 → 리뷰어 스타일 → 라운드 흐름 → 학습 포인트 → 내 답변 패턴 → 승격 후보"
- **비교 페이지** — 관련 개념 비교 (예: `비교_SQL_vs_NoSQL.md`)
- **종합 페이지** — 여러 개념을 엮은 상위 주제 (예: `컴퓨터부품 요청 흐름.md`, `overview_데이터베이스.md`)
- **리뷰 페이지** — 여러 PR(summary_PR_*)에서 반복 검출된 판단 기준·트레이드오프 원칙 (예: `review_상속_vs_합성.md`). 프레임워크: "이 코드가 왜 문제인가 → 대안 → 적용 기준 → 흔한 반론과 답 → 학습 노트"

## 페이지 작성 규칙
- 모든 페이지는 YAML frontmatter 포함: `tags`, `created`, `sources`, `related`
- 학습 프레임워크 적용: "왜 이 이름인가 → 기존 문제 → 해결 방법"
- 관련 페이지는 `[[wikilink]]`로 연결
- 새 페이지 생성 또는 기존 페이지 수정 시 반드시 `index.md`와 `log.md` 업데이트

## 파일명 prefix 규칙
- 유형 prefix(`summary_`, `review_`, `overview_`, `비교_`)는 **그룹핑·탐색 편의를 위한 도구**지 강제 규칙은 아님
- **제목 자체로 유형이 명확하면 prefix 생략 가능** (예: "컴퓨터부품 요청 흐름"은 종합 지도임이 자명 → prefix 불필요)
- prefix를 붙이는 게 이득인 경우: 원천 소스와 1:1 매핑되는 유형(`summary_`, `summary_PR_`), 동일 패턴이 여러 개 생기는 유형(`review_`)
- prefix를 생략해도 되는 경우: 단독으로 존재하는 종합 페이지, 한글 제목 자연스러움이 더 중요한 경우

## 기존 정리본 Ingest 규칙
- 기존 정리본(raw/notes/)은 형식이 다양할 수 있음
- 원본은 그대로 보존하고, wiki/에 새 형식으로 변환하여 작성
- 변환 시 학습 프레임워크와 frontmatter를 적용

## 특수 파일
- `index.md` — 전체 위키 목차 (카테고리별 정리)
- `log.md` — 작업 이력 (시간순 append-only)
