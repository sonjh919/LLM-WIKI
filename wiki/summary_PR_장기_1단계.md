## 주요 학습 포인트 (이 PR에서 새롭게 드러난 것)

### 1. DTO 필요성 판단 — 콘솔 vs 웹
- 나의 초기 논거: "view가 domain을 모르게 하기 위해" + "출력 최적화 데이터"
- 재고 후 결론: **view가 domain을 몰라야 한다는 원칙은 HTTP 통신하는 웹 환경에서 더 중요**
- 콘솔 앱에선 DTO 비용(중복 객체, 변환 로직)이 혜택보다 클 수 있음 → **삭제 선택**
- 교훈: **원칙은 맥락 위에서 작동한다**. "항상 맞는 것"처럼 보이는 원칙도 적용 비용을 저울질해야 함

### 2. MVC 패턴도 맥락 의존 — "웹의 특수성"
- 나의 질문: Application이 흐름을 담당하면 Controller는 왜 필요한가?
- seovalue 답: **"Owner는 링크"** — 스스로 선택해서 저를 설득시키세요
- 교훈: 설계에 단일 정답 없음. **자기 방식을 언어화해 방어할 수 있으면 그게 답**
- 학습 방법: 직접 두 방식 비교 구현 (블랙잭 PR의 메타 원칙과 겹침)

### 3. 추상 클래스 설계 의도 — 세 번째 반복
- 블랙잭에선 "일반 클래스 상속 지양"
- 장기에선 **"추상 클래스로 만든 이유가 뭐예요? 왜 team/position/routes가 필수 요소죠?"**
- 내 답: **"타입 계층을 나누어 기능을 확장시키기 위한 용도"** (NormalPiece/JumpingPiece/StraightPiece 중간 계층)
- seovalue 재질문: "**일반/점프의 속성은 Piece에 담겨야 할까요?**" — 중간 타입 계층 도입의 정당성 재고
- 교훈: 추상화의 목적이 "확장"일 때도 **그 확장이 실제로 발생할지** 한번 더 질문

### 4. protected의 의미 — 캡슐화 시각에서
- seovalue 질문: "protected로 선언한 이유가 궁금해요. protected는 어떤 역할을 하나요?"
- 내 재고: 단순히 "상위 필드를 하위에서 쓰려고" → **캡슐화 침해, 응집도 저하**
- 적용: 모든 필드 private + 각 하위 클래스가 자기 필드 정의 + getter 재정의
- 새 불편: getter를 추상 메서드로 정의하고 하위마다 재정의 → 중복
- **미해결 트레이드오프**: protected의 편의 vs private+getter 재정의 중복 vs Piece에 공통 필드 두기

### 5. "어차피 보드를 호출하는데 굳이 Piece 내부에서" — 책임 경계
- 내 설계: `piece.possibleRoutes(board)` — piece가 자기 이동 규칙을 알고 board 상태를 본다
- seovalue 질문: 왜 board 쪽이 아니라 piece 쪽에서 수행하나?
- 핵심 논점: **"자기 정보는 자기가 안다"** vs **"상태를 가진 쪽이 계산한다"** — 두 관점 모두 성립
- 교훈: 책임 경계 결정은 메시지 방향의 선택 — 정답보다 **일관성 있는 근거**

### 6. 매번 생성 vs 재사용 — 객체 생명주기
- 내 원래 의도: 입력받을 때마다 새 PositionConverter → 작업 후 소멸
- seovalue 제안: PositionConverter는 **상태 없는 유틸성 객체** → 재사용
- 교훈: **상태를 안 가지는 객체는 재사용이 기본** — 메서드 파라미터로 매번 만들 필요 없음

### 7. Turn을 객체로 vs Board에 병합
- 내 고민: Turn 객체로 뺄까? → 인자로 자주 넘겨야 해서 복잡도 상승
- 내 결론: Turn 자체가 할 행동이 없음 + Piece 집합 관리하는 Board가 같이 갖는 게 자연스러움 → **Board의 필드로**
- 교훈: **행동이 없는 개념은 독립 객체로 만들 가치가 낮음** — 상태만 있다면 소속이 더 자연스러운 곳에 편입

### 8. 긴 메서드 분리의 걸림돌 — context 객체 활용
- 내 고민: 분리하려고 하면 **인자가 4~5개로 늘어나 오히려 가독성 저하**
- seovalue 제안: **JumpContext같은 Parameter Object**로 묶어 주고받기 → 분리 가능
- 교훈: "분리하면 인자가 늘어난다"는 분리를 포기할 이유가 아니라 **Parameter Object 도입 신호**

### 9. 테스트 — 각 도메인 하위에, 부정 케이스도
- 지적 1: `PieceTest` 하나에 모든 기물 테스트 → **각 Piece별 테스트 클래스로 분리**
- 지적 2: 가능한 경로만 검증 — **불가능한 경로일 때 이동하지 않는지도 검증**
- 교훈: 테스트 구조는 도메인 구조를 따르고, **성공 케이스만큼 실패 케이스가 중요**

### 10. 설계 vs TDD 중간 지점 — 방법론
- 내 고민: 완벽한 설계 후 구현 vs 설계 없이 TDD — 페어와 스타일 충돌
- seovalue 답: **"완벽한 설계보다는 전체적인 그림"** — 같은 도메인 이해를 공유하는 것이 핵심
- 교훈: 설계의 완벽함이 아니라 **페어 간 도메인 합의**가 충돌 비용을 줄인다
