## 왜 이 이름인가

Storage = 저장소. 보조기억장치(Secondary Storage)라고도 함. RAM이 "작업 공간"이라면 Storage는 "창고" — 당장 쓰지 않는 데이터를 영구적으로 보관하는 곳.

## 기존 문제

[[RAM]]은 빠르지만 휘발성이고 용량이 작다. 전원이 꺼지면 모든 데이터가 사라진다. "프로그램과 데이터를 영구적으로, 대용량으로 보관할 곳이 필요하다."

## 어떻게 해결하는가

Storage는 **비휘발성** 대용량 저장장치로, 전원이 꺼져도 데이터가 유지된다.

### HDD vs SSD

```mermaid
graph LR
    HDD[HDD<br/>기계식<br/>저렴<br/>느림]
    SSD[SSD<br/>전자식<br/>비쌈<br/>빠름]
```

- **HDD** — 회전하는 디스크 위를 헤드가 이동하며 읽기/쓰기. 순차 접근은 괜찮지만 임의 접근이 느림
- **SSD** — 반도체 칩에 전기적으로 읽기/쓰기. 임의 접근이 빠르고 물리적 충격에 강함

### RAM과의 관계

프로그램 실행 흐름: Storage에서 프로그램을 읽음 → [[RAM]]에 올림 → [[CPU]]가 처리

## 백엔드 개발에서의 활용

- **DB I/O 성능**: MySQL, PostgreSQL의 데이터 파일은 Storage에 저장됨. HDD → SSD 교체만으로 쿼리 응답 속도가 크게 개선되는 이유
- **디스크 I/O 병목**: 느린 쿼리의 원인이 SQL이 아니라 디스크 읽기일 수 있음. `EXPLAIN`으로 풀 테이블 스캔 여부를 확인하는 것이 이와 연결됨
- **로그/파일 저장**: 애플리케이션 로그, 업로드 파일 등은 Storage에 기록. 대량 로그 쓰기가 디스크 I/O를 잡아먹으면 서비스 전체가 느려질 수 있음
- **클라우드에서의 선택**: AWS EBS(gp3, io2) 같은 블록 스토리지 유형을 고를 때, IOPS와 처리량이 곧 Storage 성능 선택
