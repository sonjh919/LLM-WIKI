## 왜 이 이름인가

CPU = Central Processing Unit (중앙 처리 장치). 컴퓨터의 모든 연산과 제어가 이 한 곳에서 "중앙 집중적으로" 처리되기 때문에 붙은 이름.

## 기존 문제

초기 컴퓨터는 연산, 제어, 저장이 분리된 별도 회로였다. 회로를 바꿔야 다른 계산을 할 수 있었고, 범용적이지 못했다. "프로그램을 메모리에 저장하고 하나의 장치가 순차적으로 읽어서 실행하면 되지 않을까?" → 폰 노이만 구조의 핵심이 CPU.

## 어떻게 해결하는가

CPU는 세 가지 구성 요소가 협력하여 **어떤 명령어든** 처리한다:

```mermaid
graph LR
    Control[제어장치] --> ALU[ALU]
    ALU --> Register[레지스터]
    Register --> Control
```

- **[[제어장치]]** — 명령어를 가져와(Fetch) 해석(Decode)하고, 다른 부품에 제어 신호를 보냄
- **[[ALU]]** — 산술 연산(+, -)과 논리 연산(AND, OR)을 수행
- **[[레지스터]]** — CPU 내부 초고속 임시 저장 공간, 연산에 필요한 데이터를 즉시 사용

이 세 요소가 **[[명령어 실행 사이클]]** (Fetch → Decode → Execute)을 반복하며 프로그램을 실행한다.

## 코어와 멀티코어

**"1개 코어" = 3요소(제어장치·ALU·레지스터) 세트 하나.** 즉 F-D-E 사이클을 독립적으로 굴릴 수 있는 "처리 단위" 하나를 뜻한다.

- **1코어 CPU** — 같은 순간에 명령어 **1개**만 실행 가능
- **N코어 CPU** — 같은 순간에 명령어 **N개를 진짜 병렬로** 실행 가능 (단순 교대가 아닌 물리적 동시 실행)

> "멀티코어"는 CPU 칩 하나 안에 이 3요소 세트가 여러 개 들어 있는 구조다.

## 관련 개념

- **[[컨텍스트 스위칭]]** — 코어보다 스레드가 많을 때 OS가 스레드를 번갈아 실행하는 방식. "컨텍스트"의 실체는 **레지스터 값들의 스냅샷**이다. CPU와 OS의 교차점 개념이라 별도 페이지로 분리.
- **[[컴퓨터부품 요청 흐름]]** — CPU가 HTTP 요청 처리의 어느 단계에서 등장하는지 5대 부품 조감도

## 백엔드 개발에서의 활용

### 스레드 풀 사이징 — CPU 바운드 vs I/O 바운드

| 작업 유형 | 병목 | 스레드 풀 크기 | 이유 |
|-----------|------|---------------|------|
| **CPU 바운드** (암호화, 정렬, 이미지 처리) | CPU 연산 | **≈ 코어 수** (예: 8코어 → 8~10) | 실제 병렬 실행은 코어 수만큼뿐. 그 이상은 **컨텍스트 스위칭 오버헤드**만 늘어남 |
| **I/O 바운드** (DB 조회, 외부 API 호출) | 외부 대기 | **>> 코어 수** (예: 8코어 → 50~200) | 각 스레드가 응답 대기로 CPU를 놀리는 시간이 김. 스레드를 많이 두면 대기 중에 다른 스레드가 코어를 씀 |

### 기타

- **JIT 컴파일**: JVM이 자주 실행되는 바이트코드를 CPU 네이티브 코드로 변환. 서버 워밍업 시간이 존재하는 이유
- **비동기/논블로킹 대안**: DB 대기가 극단적으로 많은 API 서버는 스레드 풀을 키우는 대신 비동기 모델(WebFlux 등)이 더 효과적일 수 있음
