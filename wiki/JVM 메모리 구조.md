## 왜 이 이름인가

**JVM = Java Virtual Machine.** Java 코드(.class 바이트코드)를 실제 CPU에서 실행 가능한 기계어로 바꿔 돌리는 **가상 머신**. 어떤 OS에서든 같은 `.class`를 실행할 수 있게 해주는 "OS 위의 런타임".

**"JVM 메모리 구조"** = JVM이 OS에게서 할당받은 [[RAM]] 공간을 **내부적으로 역할별로 쪼개 쓰는 방식**.

## 기존 문제

Java는 OS에 직접 접근하지 않고 JVM을 거친다. 그래서 JVM이 자체적으로:
- 객체를 어디에 둘지 결정해야 함
- 메서드 호출 추적을 스스로 해야 함
- 클래스 정보(.class)를 어딘가에 캐싱해야 함

→ JVM이 받은 RAM을 **역할별 영역으로 구획**해 관리하는 구조가 필요.

## 어떻게 해결하는가

### 3개 주요 영역

| 영역 | 저장 내용 | 스레드 간 |
|------|----------|----------|
| **힙(Heap)** | `new`로 생성된 객체 | 공유 |
| **스택(Stack)** | 메서드 호출 프레임·지역 변수 | **스레드별 별도** |
| **메서드 영역(Method Area)** | .class 메타데이터·static 변수·상수 풀 | 공유 |

- **힙** — 공유 창고. `User u = new User()` 하면 참조 `u`는 스택에, 실제 `User` 객체는 힙에
- **스택** — 스레드 독립. 각 스레드가 자기 메서드 호출 순서와 지역 변수를 기록. 공유 불가
- **메서드 영역** — `.class`를 파싱한 클래스 메타데이터·static 변수. Java 8부터는 "Metaspace"라는 이름으로 네이티브 메모리에 위치

### JVM 옵션

- `-Xmx2g` — 힙 **최대** 크기 2GB 제한
- `-Xms512m` — 힙 **초기** 크기
- `-Xss1m` — 스레드당 스택 크기 (기본값 1MB 내외)

**왜 한계 설정이 필요한가**:
1. **공존** — 서버 한 대에 OS·DB·다른 서비스가 공존. JVM이 무제한이면 모두가 죽음
2. **OOM을 빨리 드러내기** — 한계 없으면 스왑까지 먹으며 전체 시스템이 느려지다 죽음
3. **컨테이너 환경** — Docker/K8s에선 컨테이너에 RAM 제한이 걸려있음. JVM이 호스트 기준으로 힙을 잡으면 **OOMKilled**

## OOM (Out of Memory)

### 전형 시나리오 3종

**1. 메모리 누수 (코드)**
```java
static List<User> cache = new ArrayList<>();  // static은 GC 대상 X
// 계속 add만 하고 remove 없음 → 힙이 영원히 채워짐
```

**2. 대용량 쿼리 한 번에 로딩 (DB)**
```java
List<Order> all = orderRepo.findAll();  // 100만 건이면 끝
```
→ 페이징(`Pageable`)·스트리밍(`Stream<Order>`) 사용

**3. Eviction 정책 없는 캐시 (캐시)**
```java
Map<String, User> cache = new HashMap<>();  // 무한 put
```
→ Caffeine/Redis처럼 크기 제한·TTL 있는 캐시 사용

### OOM 에러 메시지로 영역 구분

| 에러 메시지 | 터진 영역 | 원인 |
|------------|----------|------|
| `Java heap space` | 힙 | 객체 누적 (가장 흔함) |
| `GC overhead limit exceeded` | 힙 (GC) | 힙이 꽉 차서 GC가 제 일을 못함 |
| `Metaspace` | 메서드 영역 | 클래스 로딩 누적 (핫리로드 등) |
| `unable to create new native thread` | OS 스레드 | 스레드 생성 한계 |

## 스레드와 스택

**각 스레드는 자기만의 Stack을 가진다.** Heap은 공유하지만 Stack은 스레드별로 별도 — 스레드가 각자 독립적으로 메서드를 호출하니까 **호출 순서 메모를 공유할 수 없어서**다.

```
동시 요청 1000개 = 스레드 1000개 = Stack만 1GB (1MB × 1000)
  + 각 요청이 Heap에 만든 객체
  → RAM 소비 폭발
```

여기에 [[컨텍스트 스위칭]] 오버헤드(CPU 낭비)까지 겹쳐 **"스레드를 무한정 늘리면 안 되는 이유"** 의 두 축이 된다. 비동기/논블로킹(WebFlux, Netty)은 소수 스레드가 이벤트 루프로 다수 요청을 처리해 **Stack 소비와 스위칭 비용 모두 회피**한다.

> **Java Thread vs OS 스레드**: HotSpot JVM에선 `new Thread()` 하나가 OS 스레드 하나와 **1:1 매핑**. 즉 JVM Stack을 먹는 스레드와 CPU 컨텍스트 스위칭의 대상 스레드는 **같은 스레드**다.
>
> (예외: Java 21 가상 스레드 — OS 스레드와 1:1이 아닌 JVM 관리 경량 스레드)
