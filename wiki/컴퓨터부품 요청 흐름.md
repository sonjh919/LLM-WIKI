# 컴퓨터부품 요청 흐름 — HTTP 요청 한 개 조감도

5대 부품([[CPU]]·[[RAM]]·[[Storage]]·[[System Bus]]·[[IO Device]])이 서버의 HTTP 요청 하나를 처리하는 동안 각각 어느 순간에 등장하는지 엮은 조감도. 개별 부품 페이지가 **"무엇"**을 담는다면, 이 페이지는 **"어떻게 연결되는가"**를 담는다.

## 요청 처리 9단계

```
1. [NIC (I/O)]  HTTP 요청 수신
2. [DMA]        NIC → RAM (CPU 개입 없이 System Bus 사용)
3. [CPU]        인터럽트로 알림, RAM의 요청 데이터 읽음
4. [CPU]        라우팅·비즈니스 로직 처리 (Spring Controller 등)
5. [CPU]        DB 조회 필요 → RAM 경유 → NIC로 DB 서버에 쿼리 송신
                (DB 서버 쪽도 같은 구조 반복)
6. [Storage]    디스크 → RAM 로드 (버퍼 풀 히트면 생략)
7. [CPU]        응답 생성 → RAM 적재
8. [DMA]        RAM → NIC (송신)
9. [NIC (I/O)]  클라이언트에 응답 전송
```

## 부품 역할 요약

| 부품 | 역할 | 대표 장면 |
|------|------|----------|
| [[IO Device]] (NIC) | 외부 경계 | 1·9단계 수신·송신 |
| [[System Bus]] | 부품 간 데이터 이동 | 전 단계 기반 |
| [[DMA]] | CPU 없이 I/O ↔ RAM 직결 | 2·8단계 |
| [[RAM]] | 현재 작업 데이터 적재 | 3·6·7단계 |
| [[CPU]] | 실제 연산 | 3·4·5·7단계 |
| [[Storage]] | 영구 데이터(DB·로그) | 6단계 |

## 왜 이 조감도가 중요한가

**부품 개별 이해 ≠ 조감도 이해**. 각 부품을 알아도 "한 요청이 흐를 때 어느 순서로 등장하느냐"는 별도의 인지 작업. 이 흐름을 잡으면:

- **병목 위치 추론** — 느린 API의 원인이 CPU인지 DB I/O인지 네트워크인지 구분 가능
- **최적화 맥락** — [[DMA]]·zero-copy·버퍼 풀·[[IO Device|I/O 멀티플렉싱]]이 **어느 단계를 겨냥한 기법**인지 체감
- **확장 시 전조** — 트래픽 증가 시 어느 부품이 먼저 포화될지 예상

## 확장 버전

이 조감도는 기본형. 실무에서는 아래 레이어가 얹힘:
- **TCP/IP 스택** — NIC에서 받은 패킷이 애플리케이션에 도달하기까지의 OS 레이어
- **웹 서버 (Nginx 등)** — 톰캣·WebFlux 앞단의 리버스 프록시
- **여러 스레드·연결 동시 처리** — [[JVM 메모리 구조]]의 스레드 모델, [[컨텍스트 스위칭]]
- **캐시 레이어** — Redis 등이 DB 앞단에 들어가면 6단계가 변형됨
